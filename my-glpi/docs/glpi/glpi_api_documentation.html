Documentation API REST GLPI

Référence complète de l'API REST GLPI — deux architectures disponibles : Low-Level (v1) et High-Level (v2) avec support OAuth2, RSQL et GraphQL.

GLPI 11.0.7 API v1 — /apirest.php API v2 — /api.php
00 Vue d'ensemble

L'API REST GLPI expose toutes les ressources du système d'information via HTTP. Elle permet aux applications externes d'effectuer des opérations CRUD, des recherches avancées, des actions massives et des téléversements de fichiers.

Prérequis : Activer l'API dans Configuration > Générale > API. Définir l'URL de l'API correspondant à votre instance GLPI. Créer un client API pour obtenir un App-Token.

Deux architectures d'API

API v1 — Low-Level
Accès direct aux objets CommonDBTM. Endpoint principal : /apirest.php. Authentification par Session-Token + App-Token.
Stable / Legacy
API v2 — High-Level
Couche d'abstraction moderne. Endpoint : /api.php/v2/. OAuth2, RSQL filtering, OpenAPI/Swagger, GraphQL.
Moderne / Recommandée
Note GLPI 11 : L'API v2 est désormais disponible et recommandée pour les nouvelles intégrations. L'API v1 reste entièrement fonctionnelle. Spécifier v1 dans l'URL route vers le legacy, v2 vers la High-Level API.

03 Authentification — API v2 (OAuth2)
L'API v2 utilise exclusivement OAuth2. Créer le client OAuth dans Configuration > Clients OAuth. Chaque client génère automatiquement un client_id et un client_secret.

Scopes disponibles

ScopeDescription
apiAccès général à l'API (tous les endpoints)
userAccès aux informations de l'utilisateur connecté
emailAccès à l'email par défaut de l'utilisateur
graphqlAccès à l'endpoint GraphQL
inventorySoumission d'inventaire (Client Credentials uniquement)
statusAccès aux endpoints de statut

Grant type : Password (recommandé pour scripts)

bash — Obtenir un access_token
curl -X POST \
  -H 'Content-Type: application/json' \
  -d '{"grant_type":"password","client_id":"votre_client_id","client_secret":"votre_secret","username":"admin","password":"motdepasse","scope":"api"}' \
  'https://glpi.mondomaine.fr/api.php/token'

# Réponse 200
{
  "token_type": "Bearer",
  "expires_in": 3600,
  "access_token": "eyJ0eXAiOiJKV1QiLCJhbGciOiJSUzI1NiJ9...",
  "refresh_token": "def50200..."

09 Pagination, Tri & Filtrage

Pagination

L'API v1 utilise un paramètre range de type début-fin. Par défaut : 0-49 (50 éléments).

Exemples de pagination
# Page 1 (éléments 0 à 49)
GET /apirest.php/Ticket/?range=0-49

# Page 2 (éléments 50 à 99)
GET /apirest.php/Ticket/?range=50-99

# Éléments 200 à 249
GET /apirest.php/Ticket/?range=200-249

Les headers de réponse indiquent la plage et le total :

Content-Range: 0-49/523   # 523 tickets au total
Accept-Range: Ticket 1000   # max 1000 par requête

Tri

11 Filtrage RSQL — API v2

L'API v2 utilise RSQL (REST Query Language), un sur-ensemble de FIQL, via le paramètre filter. Bien plus lisible que les critères v1.

Syntaxe : field<opérateur>value. Combiner avec ; (AND) ou , (OR).

OpérateurSignificationExemple
==Égal àname==Jean
!=Différent destatus!=6
=in=Dans la listestatus=in=(1,2,3)
=out=Hors de la listestatus=out=(5,6)
=lt=Inférieur àpriority=lt=3
=le=Inférieur ou égalpriority=le=3
=gt=Supérieur àpriority=gt=3
=ge=Supérieur ou égalpriority=ge=4
=like=Contient (sensible)name=like=serveur
=ilike=Contient (insensible)name=ilike=SERVEUR
=isnull=Est NULLclosedate=isnull=
=isnotnull=N'est pas NULLclosedate=isnotnull=
URL — Tickets ouverts et priorité haute en v2
GET /api.php/v2/Ticket?filter=status=in=(1,2,3);priority=ge=4&sort=date&order=DESC&limit=20&offset=0
URL — Combinaison AND/OR
# AND avec ;
filter=status==1;priority=ge=4

# OR avec ,
filter=status==5,status==6

# Groupement
filter=(status==1,status==2);priority=ge=3

14 Bonnes pratiques

Gestion des sessions v1

  • Réutiliser le session_token entre les appels — ne pas ouvrir une nouvelle session à chaque requête
  • Appeler killSession à la fin du traitement pour libérer les ressources
  • En cas de 401, réauthentifier et réessayer automatiquement
  • Les sessions sont en lecture seule par défaut — ajouter session_write=true uniquement si nécessaire
  • Mode lecture seule = appels parallèles possibles. Mode écriture = appels séquentiels obligatoires

Performance et pagination

  • Toujours utiliser la pagination — ne jamais récupérer tout en une seule requête
  • Utiliser only_id=true quand seuls les IDs sont nécessaires
  • Utiliser forcedisplay en Search API pour limiter les colonnes retournées
  • Préférer la Search API à la lecture de masse pour les filtrages complexes
  • Utiliser get_sha1=true pour vérifier si un objet a changé avant de le re-télécharger

Gestion des données

  • Toujours fournir le header Content-Type: application/json
  • Les requêtes GET doivent avoir un corps vide — passer les paramètres en query string
  • Pour les champs HTML (comme content d'un ticket), encoder correctement le HTML
  • Les champs préfixés par _ (ex: _users_id_assign) sont des champs d'action spéciaux non stockés directement
  • Utiliser expand_dropdowns=true pour la lisibilité, mais travailler avec les IDs numériques en écriture

API v2 — Recommandations

  • Consulter la documentation Swagger auto-générée sur /api.php/doc
  • Utiliser le versioning explicite /api.php/v2/... pour la stabilité
  • Préférer Authorization Code pour les apps interactives, Password Grant pour les scripts
  • Implémenter le refresh token — les access tokens expirent après 1 heure par défaut
  • Explorer les capacités GraphQL sur /api.php/GraphQL pour les requêtes complexes imbriquées (lecture seule)
15 Sécurité & Permissions

Filtrage de l'accès API

Configurable dans Configuration > Générale > API :

  • Plage IPv4/IPv6 — Restreindre l'accès à des adresses IP spécifiques par client API
  • App-Token — Identifier et tracer les appels par application (chaque client a son historique)
  • Méthodes d'auth — Activer/désactiver indépendamment login/pass et token personnel

Modèle de permissions GLPI

Les appels API respectent strictement le modèle de droits GLPI :

  • Profil — Détermine les droits CRUD par type d'objet (lecture, écriture, suppression...)
  • Entité — Restreint la visibilité aux objets de l'entité active et ses sous-entités (si récursif)
  • Règles de visibilité — Un ticket privé n'est visible que par son groupe/technicien assigné
  • changeActiveProfile et changeActiveEntities permettent de basculer de contexte en cours de session
Attention : Ne jamais exposer le user_token ou le App-Token dans du code client (JavaScript front-end, dépôts publics). Utiliser des variables d'environnement côté serveur.

HTTPS obligatoire

Toujours utiliser HTTPS en production. Les tokens circulent en clair dans les headers HTTP. Une connexion non chiffrée expose les credentials de session.

Récupération de mot de passe

L'endpoint PUT /apirest.php/lostPassword permet le reset de mot de passe par email — accessible sans session, sous réserve que les notifications GLPI soient activées et que l'email appartienne à un compte existant.
